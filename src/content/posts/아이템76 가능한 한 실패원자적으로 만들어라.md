# What is “failure atomicity” ??

> After an object throws an exception, it is generally desirable that **the object still be in a well-defined, usable state, even if the failure occurred in the midst of performing an operation.**

atomic이란 각 연산이 독립적으로 수행되는 것을 말한다.
(DB에서 ACID의 Atomicity, 그것이다)
failure atomicity는 연산의 실패와 연산이 서로 독립적이라는 것을 의미한다.
**즉, 연산 도중 예외 혹은 오류가 발생하게 되면, 연산 종료 이후 연산 이전 상태값으로 유지되어야 한다.**

> Atomicity.

In computer programming, an operation done by a computer is considered atomic if it is guaranteed to be isolated from other operations that may be happening at the same time.

# Why “strive for failure atomicity” ??

그렇다면 실패원자성을 갈구해야하는 진짜 이유는 뭘까??

DB 트랜잭션의 Atomicity와의 동일한 이유이다.
**각 연산의 성공 여부에 따라 값이 변하지 않아야 하며, 이를 지킴으로써 각 연산 간 독립적으로 수행될 수 있음을 보장한다.**

만약 연산에 대한 실패에 따라 값의 상태가 변하게 된다면 — 연산에 따라 상태가 동적이라면 —
연산 간에 성공여부에 따라 각 연산의 결과가 달라지게 된다.

이는 독립성을 해친다.

따라서 우리는 연산에 따른 exception과 상관없이 인스턴스의 내부 상태값은 일정하게 유지하게끔 해야한다.

> Atomic operations are critically important when dealing with shared resources.
> In traditional programming, failure in one thread can easily affect the other one.
