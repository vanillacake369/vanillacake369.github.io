## 무엇을 공부하였는가 🤔


> 스프링은 객체 지향언어가 가진 강력한 특징인 **OOP를 살려내는 프레임워크**
> 그렇다면 좋은 OOP란 어떻게 지키는 것인가?

### OOP란?

> 컴포넌트를 쉽고 유연하게 변경할 수 있도록 함 ⇒ 다형성과 SOLID패턴이 이를 서포트

### 다형성??

그게 뭔데??

> 역할과 구현을 분리한다!

하나의 클래스에 대해 다양한 객체를 만들어 객체를 갈아끼움에 따라 다양한 Act를 할 수 있게 한다.

Ex) 자동차 ⇒ 기아,테슬라,BMW ….

이렇게 역할과 구현을 분리하면??

- 클라이언트는 구현 대상의 내부 구조를 몰라도 된다!
- 구현체의 내부 구조가 변경되어도, 클라이언트에 영향을 받지 않음

클라이언트를 변경하지 않고, 서버에서의 인터페이스에 대해 다양한 객체를 갈아끼움으로서, 다형성을 구현할 수 있다!!

![](/images/notion/faf6861961e4669e.png)

***대신!

인터페이스가 변하면?? ⇒ 클라이언트,서버 모두에 큰 변경이 발생한다***
***따라서 안정적이게 인터페이스를 설계하는 것이 중요하다.(이는 자바,스프링 뿐 아니라 API 인터페이스 자체를 설계하는 것에도!)***

### SOLID 패턴

- Single Responsibility : 한 클래스에 하나의 책임(여러 책임 X) ⇒ 코드를 변경 시, 여러 책임 변경하게 된다면 해당 클래스를 의존하는 모든 것을 수정해야함
- Open Closed : 자손 클래스를 사용하여 다형성 구현 시, 클라이언트(호출하는)의 변경사항 막기
- Liskov Substitution : * abstract class*를 사용하여 다형성 구현 시, 서브 타입을 기반 타입으로 치환할 수 있게 하기
- Interface Segregation : 최대한 여러 인터페이스로 쪼개어 구현하기.

비대한 인터페이스 지양.⇒ 비대한 인터페이스 생성 시, 서브 타입을 기반 타입으로 바꿀 수 없게 될 수 있음.

즉, LSP 위반 가능성 있음
- Dependency Inversion : 하위 레이어에 대한 인터페이스를 정의 후, 상위에서 이를 호출하여 의존성을 낮추자

기능확장 가능성이 있다면 ⇒ 설계 시, 인터페이스를 먼저 설계한 다음, 구현하자.

기능확장 가능성이 없다면 ⇒ 우선 하드코딩을 해놓고, 추후 리팩토링을 해주자.

## 왜 쓰는가 ❓


결국 다음과 같이 요약된다.

> 좋은 자바 애플리케이션 만들기 ⇒ 다형성과 SOLID패턴을 유지하며 개발 ⇒ 이를 위한 스프링의 여러 기능 사용

단 한 가지 의구심이 들었다.

다형성이 중요한 건 백만번 천만번 생각해도 맞다.

그래야 클라이언트단(호출하는 사이드) 과 별개로 서버단(기능을 서비스해주는 사이드)에 대해 독립적으로 수정 및 변경이 가능하니까
다만 SOLID 패턴과 디자인패턴이 얼마나 중요하길래 이를 지원해주기 위해 스프링에서 기능들을 제공해주는 걸까?

구글링을 해보았으나 다들 추상적인 말 뿐이라 - 변경하기 용이하고 유연하다 등등 - 직접 경험해보지 않는 이상 이해하기 힘들다고 판단되었다.

다음에는 스프링에서 제공하는 여러 기능들에 대해 실제 실습을 통해 익혀보도록 한다.
