# Why?

왜 배움?

---

---

스테이트풀셋은 여타 데몬셋과 레플리카셋/디플로이먼트와는 다르다. 
**영속화 상태를 유지**하는 컨테이너 워크로드이다.

## 스테이트풀셋 정의

- 영구 볼륨 클레임 선언 가능
- 영구 볼륨은 파드 하나가 소유 / 파드 여러개가 공유 가능

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: sample-statefulset
spec:
  serviceName: sample-statefulset
  replicas: 3
  selector:
    matchLabels:
      app: sample-statefulset
  template:
    metadata:
      labels:
        app: netchecker
    spec:
      containers:
        - name: nginx-container
          image: nginx
          volumeMounts:
            - mountPath: /usr/share/nginx/html
              name: www
  volumeClaimTemplates:
    - metadata:
        name: www
      spec:
        accessModes:
          - ReadWriteOnce
        resources:
          requests:
            storage: 1G
```

### spec.template.spec.containers.volumeMounts

```yaml
spec:
  ,,,
  template:
	  ,,,
    spec:
      containers:
        - name: nginx-container
          image: nginx
          volumeMounts:
            - mountPath: /usr/share/nginx/html
              name: www
```

> www 이름의 볼륨을 컨테이너의 /usr/share/nginx/html 에 마운트한다

- name : 볼륨의 이름을 지정
- mountPath : 컨테이너의 마운트 경로를 지정

### spec.volumeClaimTemplates

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: sample-statefulset
spec:
	,,,
  volumeClaimTemplates:
    - metadata:
        name: www
      spec:
        accessModes:
          - ReadWriteOnce
        resources:
          requests:
            storage: 1G
```

- `volumeClaimTemplates`선언 시 아래 조건에 부합하면  [PersistentVolumeClaim](https://kubernetes.io/docs/concepts/storage/persistent-volumes/) 를 생성하게 된다.

## 스테이트풀셋 스케일링

스테이트풀셋 또한 `spec.replicas` 를 통해 파드 수를 조정한다.

따라서 deployment/replicaset 과 같이 스케일링을 할 수 있다.

다만 deployment/replicaset 와 다른 점은 파드를 하나씩 생성하고 삭제한다.

이로 인해 스케일링 시간이 오래 걸린다.

또한 deployment/replicaset 은 무작위로 삭제되는 반면, 스테이트풀셋은 LIFO 순으로 가장 먼저 생성된 0번째 파드가 삭제된다.

이러한 패턴을 통해 마스터 노드 기반 이중화 구조 ( Master - Slave ) 에 적합하다.

## 스테이트풀셋 정책

하나씩 파드를 생성하며 Read 상태가 되면 다음 파드를 생성한다.

이는 `spec.podManagementPolicy` 의 기본값이 OrderedReady 이기 때문이다.

다만 `spec.podManagementPolicy` 정책을 Parallel 로 조장하면 deployment/replicaset 와 같이 병렬로 생성/삭제 할 수 있다.

## 스테이트풀셋 업데이트

Deployment/Daemonset 과 마찬가지로 `OnDelete`(삭제 후 새로운 파드에 적용)/`RollingUpdate`(즉시 모든 파드에 적용) 을 따른다.

다만 데이터 영속화를 따르기 때문에 `RollingUpdate` 에 따라 동시 정지/업데이트를 할 수 없다.

스테이트풀셋은 `spec.podManagementPolicy` 가 Parallel 로 설정되어있더라도 병렬로 처리되지 않고 하나씩 업데이트가 이루어진다.

또한 스테이트풀셋은 국한적 업데이트를 처리할 수 있다.

아래와 같이 spec.updateStrategy.rollingUpdate.partition 에 갯수를 지정하면 특정 갯수의 파드만 새로운 수정사항이 적용되고, 나머지는 적용되지 않는다.

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: sample-statefulset
spec:
  updateStrategy:
    type: RollingUpdate
    rollingUpdate:
      partition: 3
  serviceName: sample-statefulset
  replicas: 3
  selector:
    matchLabels:
      app: sample-statefulset
  template:
    metadata:
      labels:
        app: sample-statefulset
    spec:
      containers:
        - name: nginx-container
          image: nginx
          # What the heck is this ???
          volumeMounts:
            - mountPath: /usr/share/nginx/html
              name: www
  volumeClaimTemplates:
    - metadata:
        name: www
      spec:
        accessModes:
          - ReadWriteOnce
        resources:
          requests:
            storage: 1G  
```
