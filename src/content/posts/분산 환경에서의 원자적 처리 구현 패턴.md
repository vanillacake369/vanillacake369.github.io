# 분산 애플리케이션 데이터 일관성 패턴 비교

## 핵심 문제: Dual Write Problem

모든 패턴이 공통적으로 해결하려는 문제는 **"DB 저장과 메시지 발행을 어떻게 원자적으로 처리할 것인가"** 입니다.

```
// 문제 상황
@Transactional
void createOrder(Order order) {
    orderRepository.save(order);      // 1. DB 저장 성공
    kafka.send(new OrderCreatedEvent()); // 2. 여기서 실패하면?
    // → DB에는 저장됐지만 이벤트는 유실 = 데이터 불일치
}

```



> 💡 모듈로 다 나눠져있을 때 어떻게 서비스 디스커버리 할 수 있을까?



## 1. Transactional Outbox Pattern (with Kafka)

### 처리 방법

```
┌─────────────────────────────────────────┐
│            Application                   │
│  ┌─────────────────────────────────┐    │
│  │      Single DB Transaction       │    │
│  │  ┌───────────┐  ┌─────────────┐ │    │
│  │  │  Orders   │  │   Outbox    │ │    │
│  │  │   Table   │  │   Table     │ │    │
│  │  └───────────┘  └─────────────┘ │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
                    │
                    ▼ (Polling or CDC)
              ┌───────────┐
              │   Kafka   │
              └───────────┘

```
```java
@Transactional
void createOrder(Order order) {
    orderRepository.save(order);
    // 같은 트랜잭션 내에서 Outbox 테이블에 이벤트 저장
    outboxRepository.save(new OutboxEvent(
        "OrderCreated",
        toJson(order)
    ));
}

// 별도 프로세스: Outbox Relay
@Scheduled(fixedDelay = 100)
void relayEvents() {
    List<OutboxEvent> events = outboxRepository.findUnpublished();
    for (OutboxEvent event : events) {
        kafka.send(event);
        event.markAsPublished();
    }
}

```

### 해결하고자 한 것

DB 쓰기와 메시지 발행의 원자성 보장 (At-least-once delivery)

### 특징

- 로컬 트랜잭션만 사용하여 단순함
- 메시지 순서 보장 가능 (Outbox 테이블에 sequence 추가)

### 단점

- Polling 방식은 지연 발생 및 DB 부하
- Outbox 테이블 관리 필요 (정리, 인덱싱)
- Relay 프로세스의 고가용성 보장 필요

### 적합한 유스케이스

- 중소규모 시스템에서 간단하게 이벤트 발행이 필요한 경우
- 실시간성 요구가 크지 않은 경우



## 2. CDC (Change Data Capture)

### 2-1. Debezium

```
┌──────────┐    ┌──────────────┐    ┌───────────┐    ┌──────────┐
│ Database │───▶│   Debezium   │───▶│   Kafka   │───▶│ Consumer │
│  (WAL)   │    │  Connector   │    │           │    │          │
└──────────┘    └──────────────┘    └───────────┘    └──────────┘
     │
     └── PostgreSQL: WAL (Write-Ahead Log)
         MySQL: Binlog
         MongoDB: Oplog

```

### 2-2. PostgreSQL Logical Replication Slot

```sql
-- Replication Slot 생성
SELECT pg_create_logical_replication_slot('my_slot', 'pgoutput');

-- 변경 사항 읽기
SELECT * FROM pg_logical_slot_get_changes('my_slot', NULL, NULL);

```

### 2-3. DynamoDB Streams & Lambda

```
┌──────────────┐    ┌──────────────┐    ┌────────┐
│   DynamoDB   │───▶│   DynamoDB   │───▶│ Lambda │──▶ SQS/SNS/Kafka
│    Table     │    │   Streams    │    │        │
└──────────────┘    └──────────────┘    └────────┘

```

### 해결하고자 한 것

- Outbox Polling의 지연과 DB 부하 문제 해결
- 애플리케이션 코드 변경 없이 이벤트 캡처

### 특징

- DB 로그 기반으로 변경 사항을 실시간에 가깝게 캡처
- 애플리케이션과 완전히 분리된 인프라 레벨 솔루션
- 모든 테이블 변경을 캡처 가능

### 단점

- 인프라 복잡도 증가 (Kafka Connect, Debezium 운영)
- DB 스키마가 곧 이벤트 스키마가 됨 (강한 결합)
- Replication Slot 관리 실패 시 WAL 축적으로 디스크 풀 위험
- 비즈니스 이벤트가 아닌 "데이터 변경"을 캡처하므로 의미론적 손실

### 적합한 유스케이스

- 레거시 시스템에서 이벤트 발행이 필요한 경우
- 대규모 시스템에서 실시간에 가까운 이벤트 스트리밍 필요
- 데이터 동기화, 캐시 무효화, 검색 인덱스 업데이트



## 3. 2PC / XA Transaction

### 처리 방법

```
        ┌─────────────────────────────────────────┐
        │         Transaction Coordinator          │
        └─────────────────────────────────────────┘
                          │
         ┌────────────────┼────────────────┐
         ▼                ▼                ▼
    ┌─────────┐      ┌─────────┐      ┌─────────┐
    │ DB (RM) │      │ MQ (RM) │      │ DB (RM) │
    └─────────┘      └─────────┘      └─────────┘

Phase 1: Prepare (모든 참여자에게 커밋 가능 여부 확인)
Phase 2: Commit/Rollback (모든 참여자가 OK면 커밋, 아니면 롤백)

```
```java
@Transactional  // JTA Transaction Manager
void transfer(String from, String to, BigDecimal amount) {
    // XA Resource 1: Oracle DB
    accountRepository.debit(from, amount);

    // XA Resource 2: PostgreSQL DB
    accountRepository.credit(to, amount);

    // XA Resource 3: JMS Queue
    jmsTemplate.send(new TransferEvent(...));

    // → 모두 성공하거나 모두 롤백
}

```

### 해결하고자 한 것

여러 리소스(DB, MQ 등)에 걸친 **강한 일관성(Strong Consistency)** 보장

### 특징

- ACID 보장
- 분산 환경에서 원자성 제공
- 표준화된 프로토콜 (JTA/XA)

### 단점

- **성능 저하**: 동기 블로킹, 락 오래 유지
- **가용성 감소**: 하나의 참여자 장애 시 전체 트랜잭션 블록
- **확장성 제한**: NoSQL, 대부분의 클라우드 서비스 미지원
- **Coordinator 장애 시 In-doubt 트랜잭션 문제**

### 적합한 유스케이스

- 금융 시스템에서 절대적 일관성이 필요한 경우
- 참여자 수가 적고 지연 허용 가능한 경우
- 모놀리식 또는 제한된 마이크로서비스 환경



## 4. TCC (Try-Confirm-Cancel)

### 처리 방법

```
┌───────────────────────────────────────────────────────────────┐
│                        TCC Coordinator                         │
└───────────────────────────────────────────────────────────────┘
        │              │              │
        ▼              ▼              ▼
   ┌─────────┐    ┌─────────┐    ┌─────────┐
   │Service A│    │Service B│    │Service C│
   └─────────┘    └─────────┘    └─────────┘

1. Try Phase:    리소스 예약 (재고 차감 → 예약 상태로)
2. Confirm:      예약 확정 (예약 → 확정)
3. Cancel:       예약 취소 (예약 → 원복)

```
```java
// 재고 서비스 TCC 구현
public class InventoryTccService {

    // Try: 재고 예약
    void tryDeduct(String itemId, int qty, String txId) {
        inventory.reserved += qty;
        inventory.available -= qty;
        reservations.put(txId, new Reservation(itemId, qty));
    }

    // Confirm: 예약 확정
    void confirm(String txId) {
        Reservation r = reservations.remove(txId);
        inventory.reserved -= r.qty;
        inventory.sold += r.qty;
    }

    // Cancel: 예약 취소
    void cancel(String txId) {
        Reservation r = reservations.remove(txId);
        inventory.reserved -= r.qty;
        inventory.available += r.qty;
    }
}

```

### 해결하고자 한 것

2PC의 긴 락 문제 해결, 비즈니스 로직 레벨에서 보상 트랜잭션 구현

### 특징

- 리소스 락 시간 최소화 (Try에서 예약만)
- 비즈니스 로직으로 일관성 관리
- 2PC보다 유연함

### 단점

- **높은 구현 복잡도**: 모든 서비스가 Try/Confirm/Cancel 구현 필요
- **비즈니스 침투적**: 도메인 모델에 TCC 개념이 스며듦
- **Cancel 실패 처리 복잡**: 멱등성 보장 필요
- **예약 상태 관리 오버헤드**

### 적합한 유스케이스

- 짧은 트랜잭션, 빠른 응답 필요
- 항공권/호텔 예약 시스템 (자연스러운 Try-Confirm 모델)
- 자원 예약이 비즈니스에 자연스러운 도메인



## 5. Saga Pattern

### 처리 방법
### Choreography (이벤트 기반)

```
Order Service ──(OrderCreated)──▶ Payment Service
                                        │
Payment Service ──(PaymentCompleted)──▶ Inventory Service
                                              │
Inventory Service ──(InventoryReserved)──▶ Shipping Service

// 실패 시 보상 이벤트 역방향 전파
Inventory Service ──(InventoryFailed)──▶ Payment Service ──(RefundIssued)──▶ ...

```

### Orchestration (중앙 조율자)

```
┌─────────────────────────────────────────┐
│            Saga Orchestrator             │
│                                         │
│  Step 1: createOrder()                  │
│  Step 2: processPayment()               │
│  Step 3: reserveInventory()             │
│  Step 4: scheduleShipping()             │
│                                         │
│  Compensation:                          │
│  - cancelShipping()                     │
│  - releaseInventory()                   │
│  - refundPayment()                      │
│  - cancelOrder()                        │
└─────────────────────────────────────────┘

```
```java
// Orchestration 예시 (Axon Framework, Temporal, etc.)
@Saga
public class OrderSaga {

    @StartSaga
    @SagaEventHandler(associationProperty = "orderId")
    void on(OrderCreatedEvent event) {
        commandGateway.send(new ProcessPaymentCommand(event.orderId, event.amount));
    }

    @SagaEventHandler(associationProperty = "orderId")
    void on(PaymentCompletedEvent event) {
        commandGateway.send(new ReserveInventoryCommand(event.orderId, event.items));
    }

    @SagaEventHandler(associationProperty = "orderId")
    void on(PaymentFailedEvent event) {
        // 보상 트랜잭션
        commandGateway.send(new CancelOrderCommand(event.orderId));
    }
}

```

### 해결하고자 한 것

- 장기 실행 트랜잭션 지원
- 분산 환경에서 최종 일관성(Eventual Consistency) 달성
- 서비스 자율성 유지

### 특징

- 로컬 트랜잭션의 체인
- 실패 시 보상 트랜잭션으로 롤백
- 서비스 간 느슨한 결합 가능

### 단점

- **복잡한 보상 로직**: 모든 단계에 보상 트랜잭션 필요
- **일관성 윈도우**: 중간 상태가 외부에 노출될 수 있음
- **디버깅 어려움**: 특히 Choreography에서 흐름 추적 복잡
- **격리성 부재**: Dirty Read, Lost Update 가능

### 적합한 유스케이스

- 마이크로서비스 간 비즈니스 프로세스
- 장기 실행 워크플로우 (주문→결제→배송)
- 최종 일관성 허용 가능한 시스템



## 6. Idempotent Consumer (BloomFilter/BitMap)

### 처리 방법

```
┌──────────┐        ┌──────────────────────────────────┐
│  Kafka   │───────▶│         Consumer                  │
│          │        │  ┌────────────────────────────┐  │
└──────────┘        │  │   Deduplication Check      │  │
                    │  │                            │  │
                    │  │  ┌──────────┐ ┌─────────┐ │  │
                    │  │  │  Bloom   │ │  Exact  │ │  │
                    │  │  │  Filter  │ │  Store  │ │  │
                    │  │  │ (1st)    │ │ (2nd)   │ │  │
                    │  │  └──────────┘ └─────────┘ │  │
                    │  └────────────────────────────┘  │
                    └──────────────────────────────────┘

```
```java
public class IdempotentConsumer {
    private final BloomFilter<String> bloomFilter;  // 빠른 1차 필터
    private final Set<String> processedIds;          // 정확한 2차 체크 (Redis, DB)

    void consume(Message message) {
        String messageId = message.getId();

        // 1차: BloomFilter (False Positive 가능, False Negative 불가)
        if (bloomFilter.mightContain(messageId)) {
            // 2차: 정확한 체크
            if (processedIds.contains(messageId)) {
                log.info("Duplicate message, skipping: {}", messageId);
                return;
            }
        }

        // 처리
        processMessage(message);

        // 처리 완료 기록
        bloomFilter.put(messageId);
        processedIds.add(messageId);
    }
}

```

### BitMap 방식 (순차 ID인 경우)

```java
// Redis BitMap 활용
public boolean isProcessed(long messageId) {
    return redis.getbit("processed_messages", messageId);
}

public void markProcessed(long messageId) {
    redis.setbit("processed_messages", messageId, 1);
}
// 10억 개 메시지 = ~125MB 메모리

```

### 해결하고자 한 것

At-least-once 메시징 환경에서 중복 처리 방지 (Exactly-once Semantics 달성)

### 특징

- BloomFilter: 메모리 효율적, 확률적 자료구조
- BitMap: 순차 ID에서 극도로 효율적
- 2단계 체크로 정확성과 성능 모두 확보

### 단점

- **BloomFilter**: False Positive 존재, 크기 조절 필요
- **저장소 관리**: TTL 설정, 오래된 ID 정리 필요
- **분산 환경**: 동시성 제어 필요 (Race Condition)
- **메시지 ID 의존**: 의미 있는 Idempotency Key 설계 필요

### 적합한 유스케이스

- 결제 처리 (중복 결제 방지)
- 재시도가 빈번한 메시징 시스템
- At-least-once → Effectively-once 변환 필요 시



## 7. Sidecar-based Outbox Relay (Envoy, Dapr)

### 처리 방법

```
┌─────────────────────────────────────────────────────────┐
│                         Pod                              │
│  ┌─────────────────┐         ┌─────────────────┐        │
│  │   Application   │         │     Sidecar     │        │
│  │                 │         │   (Dapr/Envoy)  │        │
│  │  ┌───────────┐  │         │                 │        │
│  │  │  Outbox   │◀─┼────────▶│  Outbox Relay   │────────┼──▶ Kafka
│  │  │  Table    │  │         │                 │        │
│  │  └───────────┘  │         │  - Polling      │        │
│  │                 │         │  - CDC          │        │
│  └─────────────────┘         │  - Retry        │        │
│           │                  │  - Ordering     │        │
│           ▼                  └─────────────────┘        │
│        Database                                          │
└─────────────────────────────────────────────────────────┘

```

### Dapr Outbox 예시

```yaml
# Dapr Component Configuration
apiVersion: dapr.io/v1alpha1
kind: Component
metadata:
  name: orderpubsub
spec:
  type: pubsub.kafka
  metadata:
    - name: outboxPublishPubsub: "kafka"
    - name: outboxPublishTopic: "orders"
    - name: outboxDiscardWhenMissingState: false

```
```java
// 애플리케이션 코드 - Outbox 로직 없음!
@PostMapping("/orders")
void createOrder(@RequestBody Order order) {
    // Dapr State Store에 저장하면
    // Sidecar가 자동으로 이벤트 발행
    daprClient.saveState("statestore", order.getId(), order);
}

```

### 해결하고자 한 것

- Outbox 패턴의 Relay 로직을 애플리케이션에서 분리
- 인프라 관심사의 표준화 및 재사용

### 특징

- 애플리케이션 코드 단순화 (비즈니스 로직에 집중)
- 언어/프레임워크 독립적
- 재시도, 순서 보장 등 복잡한 로직을 Sidecar가 처리
- 관찰 가능성(Observability) 내장

### 단점

- **운영 복잡도**: Sidecar 자체의 관리 필요
- **리소스 오버헤드**: Pod 당 추가 컨테이너
- **네트워크 홉 증가**: App ↔ Sidecar 간 통신
- **벤더 종속**: Dapr 등 특정 플랫폼에 의존
- **디버깅 어려움**: 문제 발생 시 추적 포인트 증가

### 적합한 유스케이스

- Kubernetes 환경의 마이크로서비스
- 다양한 언어/프레임워크 혼용 환경
- 인프라 표준화가 중요한 대규모 조직

## 패턴 비교 요약표

| 패턴 | 일관성 | 복잡도 | 성능 | 적합한 상황 |
| --- | --- | --- | --- | --- |
| **Tx Outbox** | Eventual | 낮음 | 중간 | 단순한 이벤트 발행 |
| **CDC** | Eventual | 중간 | 높음 | 레거시 통합, 실시간 동기화 |
| **2PC/XA** | Strong | 중간 | 낮음 | 강한 일관성 필수 (금융) |
| **TCC** | Strong* | 높음 | 중간 | 자원 예약 도메인 |
| **Saga** | Eventual | 높음 | 높음 | 장기 실행 비즈니스 프로세스 |
| **Idempotent Consumer** | - | 낮음 | 높음 | 중복 처리 방지 (보조 패턴) |
