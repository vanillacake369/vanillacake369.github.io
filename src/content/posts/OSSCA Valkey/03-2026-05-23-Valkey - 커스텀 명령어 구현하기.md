# Valkey 소스 코드에 커스텀 명령어 추가하기 — echoTony에서 echo2\_ prefix까지

Valkey에 새로운 명령어를 추가하려면 어디를 건드려야 할까. `PING`이나 `ECHO`처럼 단순한 명령어라도 소스 코드에 바로 함수만 추가해서는 동작하지 않는다. Valkey는 명령어 metadata를 JSON으로 분리 관리하고, 빌드 시점에 코드를 생성하며, 런타임에 hashtable로 등록하는 파이프라인을 거친다.

OSSCA 2026 과정에서 `ECHOTONY`라는 커스텀 명령어를 직접 구현하면서 이 파이프라인을 따라가 보았다. 이 글은 그 과정에서 파악한 command 등록 구조, sds 문자열 체계, addReply 응답 처리 방식을 정리한 기록이다. [다음 글](/posts/valkey-네트워크-처리과정-해부)에서는 이 명령어 함수가 호출되기까지의 네트워크 경로를 추적한다.

## Command 등록 파이프라인 🔧

Valkey는 명령어를 소스 코드에 하드코딩하지 않는다. `src/commands/` 디렉터리의 JSON 파일로 metadata를 관리하고, 빌드 시점에 Python 스크립트가 이를 C 코드로 변환한다.

파이프라인은 다음 순서로 동작한다.

1. `src/commands/*.json` — 각 명령어의 이름, arity, flags, 인자 스펙을 정의
2. `utils/generate-command-code.py` — JSON을 파싱해 `commands.def`를 생성 (`commands.c`에 `#include`되어 `serverCommandTable[]` 배열로 컴파일됨)
3. `populateCommandTable()`[^9] — 서버 시작 시 메모리의 `serverCommandTable[]`을 순회하여 hashtable에 등록
4. 요청이 들어오면 `lookupCommand()`[^2]로 명령어를 찾아 `c->cmd->proc(c)` 호출

`ECHOTONY` 명령어를 위해 추가한 JSON 파일은 다음과 같다.

```json
// src/commands/echotony.json
{
    "ECHOTONY": {
        "summary": "Returns the given string.",
        "complexity": "O(1)",
        "group": "connection", // ← 명령어가 속하는 그룹
        "since": "8.1.0",
        "arity": 2, // ← 명령어 이름 포함 인자 2개
        "function": "echoTonyCommand", // ← 실제 호출될 C 함수 이름
        "command_flags": ["LOADING", "STALE", "FAST"],
        "acl_categories": ["CONNECTION", "FAST"],
        "arguments": [{ "name": "message", "type": "string" }]
    }
}
```

`arity: 2`는 명령어 이름을 포함해 인자가 정확히 2개임을 의미한다. 음수로 지정하면 최소 개수를 나타낸다. `function` 필드가 가리키는 C 함수가 실제 명령어 로직이다.

## 1단계: 입력값을 그대로 돌려주기 🏗️

JSON으로 metadata를 정의했으니, 이제 실제 함수를 구현해야 한다. 기존 `echoCommand`를 참고했다.[^1]

```c
// src/server.c — 기존 echoCommand
void echoCommand(client *c) {
    addReplyBulk(c, c->argv[1]);  // ← argv[1]을 그대로 bulk string 응답으로 반환
}
```

`client *c` 구조에서 핵심 필드는 두 가지다.

- `c->argc` — 파싱된 인자 개수 (명령어 이름 포함)
- `c->argv` — `robj *` 배열. `argv[0]`은 명령어 이름, `argv[1]`부터 실제 인자

이 패턴을 그대로 따라 `server.h`에 함수를 선언하고 `server.c`에 구현을 추가했다.

```c
// src/server.h
void echoTonyCommand(client *c);
```

```c
// src/server.c
void echoTonyCommand(client *c) {
    addReplyBulk(c, c->argv[1]);
}
```

빌드 후 테스트하면 입력값이 그대로 돌아온다.

```
127.0.0.1:6379> ECHOTONY abc
"abc"
```

입력값을 그대로 돌려주는 것은 동작하지만, 과제 요구사항은 응답 앞에 `echo2_` prefix를 붙이는 것이었다. 기존 `robj *`를 그대로 넘기는 `addReplyBulk()`로는 문자열을 가공할 수 없다. 새로운 문자열을 동적으로 만들어야 한다.

## 2단계: echo2\_ prefix 추가 — sds 문자열 체계 🏗️

### sds란

Valkey는 C 표준 문자열 대신 `sds`(Simple Dynamic Strings)를 사용한다.[^6]

```c
// src/sds.h
typedef char *sds;
```

타입 정의만 보면 일반 `char *`와 다를 바 없다. 차이는 메모리 레이아웃에 있다.

```
[ header: len | alloc | flags ] [ buf[0] buf[1] ... buf[n] \0 ]
                                  ^
                                  sds 포인터가 가리키는 위치
```

`sds` 포인터는 header가 아닌 `buf[]`를 가리킨다. header는 `buf[]` 바로 앞에 숨겨져 있다.

이 구조 덕분에 `sdslen()`은 `strlen()`처럼 `\0`을 탐색하지 않고 header의 `len` 필드를 읽어 O(1)에 길이를 반환한다.[^7] 또한 `len`을 기준으로 데이터를 처리하므로 중간에 `\0`이 포함된 binary-safe 데이터도 다룰 수 있다.

### 구현 코드

```c
// src/server.c
void echoTonyCommand(client *c) {
    sds reply = sdsempty();                                           // ← 빈 sds 생성
    reply = sdscatfmt(reply, "echo2_%S", (sds)objectGetVal(c->argv[1])); // ← prefix + 인자 결합 [^12]
    addReplyBulkSds(c, reply);                                        // ← 응답 전송 + 메모리 해제
}
```

세 함수의 역할은 다음과 같다.

**`sdsempty()`**[^4] — 길이 0의 빈 sds를 생성한다. 내부적으로 `sdsnewlen("", 0)`을 호출한다.

**`sdscatfmt()`**[^5] — format string을 해석하며 sds 뒤에 값을 이어 붙인다. `printf` 계열과 유사하지만 sds 전용 포맷 지시자를 사용한다.

- `%S` — `sds` 타입 인자
- `%s` — C string 인자
- `%i`, `%I` — int, long long 인자
- `%u`, `%U` — unsigned int, unsigned long long 인자

`c->argv[1]`은 `robj *`이므로 `objectGetVal()`로 내부 sds 포인터를 꺼낸 뒤 `%S`에 전달했다.

**`addReplyBulkSds()`**[^3] — 생성한 sds를 RESP bulk string 형식으로 클라이언트에게 전송한다. 이 함수는 전달받은 sds의 소유권을 가져간다. 내부에서 `sdsfree()`[^10]를 호출하므로, 호출 이후 `reply`를 직접 해제하면 double-free가 발생한다.

```
127.0.0.1:6379> ECHOTONY abc
"echo2_abc"
```

## addReply 계열 함수 비교 📋

1단계에서 2단계로 넘어갈 때 `addReplyBulk()` 대신 `addReplyBulkSds()`를 선택한 이유가 여기에 있다. Valkey는 입력 타입과 메모리 소유권에 따라 여러 addReply 함수를 제공한다.

| 함수                    | 입력 타입    | RESP framing | 메모리 소유권              |
| ----------------------- | ------------ | ------------ | -------------------------- |
| `addReplyBulk()`        | `robj *`     | bulk string  | object 소유권 유지         |
| `addReplyBulkSds()`     | `sds`        | bulk string  | 호출 후 내부에서 sdsfree() |
| `addReplyBulkCBuffer()` | buffer + len | bulk string  | 소유권 유지                |
| `addReplyBulkCString()` | C string     | bulk string  | 소유권 유지                |
| `addReplyLongLong()`    | integer      | `:n\r\n`     | N/A                        |

새로 만든 sds를 응답으로 내보낼 때는 `addReplyBulkSds()`가 적합하다. 메모리 해제를 함수가 담당하므로 호출 후 별도 처리가 필요 없다. `addReplyBulk()`는 기존 `robj *`를 그대로 전달할 때 쓴다.

## 응답 버퍼까지의 내부 흐름 🔍

`addReplyBulkSds()`가 호출된다고 응답이 즉시 소켓에 쓰이는 것은 아니다. 응답 데이터는 클라이언트 내부 버퍼에 쌓인 뒤, 이벤트 루프가 실제 전송을 담당한다.

```
addReplyBulkSds(c, reply)
    └─> prepareClientToWrite()           // [^8] 쓰기 가능 상태 확인
            └─> _addReplyToBufferOrList()
                    ├─> c->buf           // 고정 버퍼 (16KB)
                    └─> c->reply         // 넘칠 경우 linked list
    └─> sdsfree(reply)                   // sds 소유권 해제
```

`prepareClientToWrite()`[^8]는 클라이언트가 쓰기 가능한 상태인지 확인하고, 필요하면 pending write 큐에 등록한다. 이후 이벤트 루프의 `beforesleep` 단계에서 `writeToClient()`가 실제로 소켓에 쓴다.

대부분의 단순 명령어 응답은 `c->buf` 안에서 처리된다. 버퍼가 가득 차면 `c->reply` linked list로 이어진다.

## 마무리 📌

커스텀 명령어 하나를 추가하려면 세 곳을 건드린다.

1. `src/commands/echotony.json` — command metadata 정의
2. `src/server.h` — 함수 선언
3. `src/server.c` — 함수 구현

Valkey의 command 등록은 JSON → codegen → hashtable lookup 파이프라인이다. 응답은 addReply 계열 → 클라이언트 내부 버퍼 → 이벤트 루프 write 경로를 통해 전달된다.

문자열을 동적으로 가공해야 한다면 `sdsempty()` → `sdscatfmt()` → `addReplyBulkSds()` 조합이 기본 패턴이다. `addReplyBulkSds()`가 sds 소유권을 가져간다는 점만 기억하면 메모리 관리 실수를 피할 수 있다.

이 글에서는 `c->cmd->proc(c)`가 호출된 이후의 응답 처리를 다루었다. 그런데 클라이언트가 보낸 바이트가 어떻게 이 함수 호출까지 도달하는가? [다음 글](/posts/valkey-네트워크-처리과정-해부)에서 그 네트워크 경로 — 이벤트 루프, epoll, accept/read/write 세 경로 — 를 추적한다.
