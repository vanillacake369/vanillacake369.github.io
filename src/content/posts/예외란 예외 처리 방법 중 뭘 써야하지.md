# 예외란?

헤드퍼스트 자바를 보면 예외처리란 **런타임에 대한 예외 발생 **케이스를 핸들링하기 위한 좋은 방법이라고 나와있다.

> Java’s exception-handling mechanism is a clean, well-lighted way to handle “exceptional situations” that pop up at runtime;



# 예외처리 핸들링 방법 : Handle or Declare

그렇다면 예외처리 핸들링의 방법은 무엇이 있을까?
두 가지 방법이 있다 : Handle(직접 처리) vs Declare(ducking,전파)

**try-catch**는 예외를 개발자가 직접 처리하여, **callee method에서 해당 예외 발생 하는 경우, 후 처리를 caller 단에서 직접 해야하는 경우**이다.
ducking은 경우가 다르다. throws를 통해 callee가 예외를 던졌고, 이를 caller가 ducking하였다면 - **즉, try-catch를 통해 예외에 대한 처리를 하지 않았다면 **- 해당 예외는 **다음 caller에게 propagate되며, 최종적으로 main에 전달된다.** 만약 main에서도 ducking을 한다면 JVM이 강제종료된다.

## Handle vs Declare(ducking)

> If you’re prepared to handle the exception, wrap the call in a try/catch, and put your exception handling/recovery code in the catch block.
> 
> Handle or Declare. It’s the law.

## 그렇다면 뭐가 더 좋을까?

> **`try-catch vs throws`**


하지만 두 패턴 모두 필요하다. 왜냐하면 caller와 callee 관점에서 사용되기 때문이다. 
callee가 예외를 던질 수 있음을 컴파일러에게 알리고, 
이에 대해 caller가 ducking, 즉 전파를 하여 다음 caller에게 전달을 할 것인지, 
아니면 caller가 try-catch를 통해 예외처리를 할 것인지 
결정해주도록 한다.

> No, both patterns are needed. If your method can handle the exception, use the second pattern, if not, use the first one to notifiy the caller.

# 예외를 처리할 때는 가장 specific exception handling 

추가적으로 예외를 처리할 때는 가장 specific exception handling 하는 게 좋다.
아래와 같은 boiler plate는 피하자.

```java
// ❌
try{
	// things could throw exception
}catch(Exception e){
	// catch by Exception
}
```
