# Intro :: 데이터가 균등하게 배분되지 않을 때의 문제점

우리는 클라이언트-서버 시스템을 사용한다.

서버를 통해 데이터를 저장하고 제공한다.

이 때, 데이터가 여러 서버에 균등적으로 배분이 되지 않을 때의 문제점은 무엇일까?

데이터가 한 서버에 밀집되어있다면 다음과 같은 문제점이 발생한다.

- SPOF 제공 가능
- 서버 확장의 제한점

_왜 데이터 균등배분을 하는 것이냐?_ 라고 묻는다면,

아래와 같이 요약할 수 있을 것이다.

- 목적(For) : SPOF 와 유연한 트래픽 처리
- 해야할 것(ToDo) : 수평적 규모 확장
- 수평적 확장을 위해 달성되어야 할 것 : 데이터 균등 배분

그렇다면 어떻게 균등 배분을 지원할 수 있을까?

천천히 살펴보자.

> 본 포스트에서는 _"데이터를 여러 서버에 균등적으로 배분"_ 에 의의를 두고 있다.
>
> _샤딩, 즉 "데이터를 여러 DB 내 균등적으로 배분"_ 과는 다른 맥락이다.

# 해시를 사용하여 분배하기

해시를 사용하여 분배하는 방식과 그에 대한 문제점을 알아보자.

일반적인 해시함수는 mod 연산, 즉 나머지 연산자를 사용한다.

아래의 해시 함수를 사용하여 N개의 서버에 데이터를 균등적으로 배분해보자.

```
serverIndex = hash(key) % N

* N은 서버의 개수
```

![](/images/velog/6fe01f4dba1a3bfc.png)

이 때 어떤 문제점이 생길 수 있을까?

바로 서버추가와 삭제처리에 따라 해시값이 변동된다는 것이다.

key 값은 추가와 삭제에 따라 동적으로 변경된다.

또한 serverIndex 값은 key 값에 따라 변경된다.

이 말인 즉슨 서버 추가와 삭제에 따라 데이터가 저장되는 서버의 위치가 변경된다는 것이다.

이를 해결하기 위해 도입된 것이 안정해시이다.

# 안정해시 개념

안정 해시는 대량의 캐시 미스 발생 가능성을 최대한 낮추기 위해 고안된 개념이다.

<span style="color:yellowgreen">**여기서 안정 해시는 일반해시의 mod 연산을 사용하지 않는 것이 주요 포인트이다.**</span>

코어 아이디어는 다음과 같다.

![](/images/velog/27bf9ec3c20f7214.webp)

- 해시 공간의 시작과 끝을 이어서 일종의 고리(ring)을 만든다.
- 해시 공간 위에 노드들을 배치한다.
- 노드들은 각각의 해시 공간을 할당받고 트래픽을 기다리게 된다.
- 이 때, 데이터에 대해 해시함수를 적용하여 링 위에서 가장 가까운 노드를 향해 라우팅한다.
- B 노드는 이후 같은 요청이 들어올 때를 대비해서 key1 을 캐시해둔다.

> 안정해시의 장점은 노드의 목록이 변하더라도 데이터 miss 확률이 적다는 것에 있다.

## 노드가 추가/제거되었을 때

![](/images/velog/00092e85397da880.webp)

E 노드가 추가되었다면 해당 노드에 재배치 되는 키 이외에는 키가 변동되지 않는다.

즉, D 와 C 노드 사이에 배치되던 키 중 일부는 새로운 E 노드로 향하게 되면서 캐시가 미스하게 되지만, 나머지 노드들은 캐시 미스가 발생하지 않는다.

기존에 키가 추가되는 경우 모든 해시값이 변동되는 것을 고려한다면, 안정해시의 부분적 재배치는 상대적인 장점이 된다.

이는 노드가 삭제되었을 때도 마찬가지이다.

![](/images/velog/305f533578991802.webp)

C 노드가 제거 되었을 때, C 노드로 향했던 키들이 D 노드로 향하여 캐시 미스가 발생하지만 그 외 공간에 배치된 키들은 캐시 미스가 발생하지 않는다.

# 안정 해시의 문제점

## 1.

전체 파티션에 대해 균등하게 배분하기가 어렵다.

![](/images/velog/36f6667c6a44ad39.webp)

서버가 추가되거나 삭제되는 상황을 감안하면 파티션(partition)의 크기를 균등하게 유지하는 게 불가능하다. \*파티션 : 인접한 서버 사이의 해시 공간

해시 값에 따라 배치되므로 전체 해시 공간을 균등하게 배분하기가 어렵다.

## 2.

데이터를 균등하게 배분하기가 어렵다.
![](/images/velog/3a832136d02f2c84.webp)

이는 첫 번째 이슈로 인해 생기는 문제이다.

위 그림에서 볼 수 있듯이 D의 파티션(해시공간)이 A,B 보다 넓다.

따라서 담당영역이 넓은 D 서버에 데이터가 몰리는 이슈가 발생할 수 있다.

데이터가 몰리면 안 좋은 이유는 명확하다.

몰리면 현상, 즉 핫스팟이 발생하면 부하가 증가되어 장애를 발생할 수 있기 때문이다.

# 가상 노드

**"전체 파티션에 대한 불균형 배분" 이슈를 해결**하기 위해 도입된 것이 바로 **"가상 노드"**이다.

**아이디어**는 간단하다.

**파티션을 잘게 잘게 쪼개고, 이를 균등하게 배분하는 식으로 해결하는 것**이다.

그렇다면 이 행위를 어떻게 처리할 수 있을까?

바로 **파티션을 가상노드라는 단위로 잘게 쪼갠 뒤, 가상노드 N개가 실제 물리서버노드를 가르키게 하는 것**이다.

![](/images/velog/f7316648a2d73dcb.png)

가상 노드는 내부적으로 물리 노드의 해시값을 가리키고 있다.

일종의 분신술(…?)이라고 생각하면 된다.

본체인 물리 노드는 해시 링 위에 배치되지 않고, 복제된 가상 노드들만이 해시 링 위에서 트래픽을 기다리고 있다.

트래픽이 가상 노드에 할당되면 내부에 존재하는 실제 노드의 해시 값으로 라우팅되는 원리이다.
