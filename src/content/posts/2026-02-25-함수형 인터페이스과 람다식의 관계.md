# 함수형 인터페이스란 무엇인가? 🤔

함수형 인터페이스란 무엇일까?? 우선 함수형 인터페이스를 사용하기 전에 인터페이스가 무엇이며 왜 써야하는지부터 간략히 알아보자.

<details>
<summary>인터페이스란?</summary>

- 인터페이스 개념

함수형 인터페이스란 아래와 같다.

- 추상 메소드를 딱 하나만 가지고 있는 인터페이스 ( = SAM (Single Abstract Method) 인터페이스 )
- [@FuncationInterface 애노테이션](https://docs.oracle.com/javase/8/docs/api/java/lang/FunctionalInterface.html) 사용

```java
@FunctionalInterface
public interface Operator{
	public int operate(int x,int y);
}
```

# 왜 쓰는가 ❓

그렇다면 함수형 인터페이스는 왜 쓰는가? 함수형 인터페이스는 함수형 프로그래밍을 하기 위해 쓰인다. 즉, 함수를 일급객체(First-class Object)처럼 다룰 수 있게 한다.

<details>
<summary>일급객체(First-class Object)</summary>

- 변수에 할당(assignment)할 수 있다.
- 다른 함수를 인자(argument)로 전달 받는다.
- 다른 함수의 결과로서 리턴될 수 있다.
</details>

함수를 일급개체로 다루면 무엇이 좋은가?

1. 고차함수(High-Order Function)을 만들 수 있다. 즉, 다른 함수를 인자로 받거나, 함수 자체를 반환할 수 있다.
2. 행위 또한 값으로 취급할 수 있다.

여기까지는 좀 이론적인 이야기고 본질적으로는 다음과 같다. 인터페이스 자체를 객체화하여 함수형 프로그래밍을 할 수 있게 없나에 대한 고민이다.

> **추상메서드 묶음인 인터페이스를 그대로 객체화하여 쓰고 싶을 때는 어떻게 해야할까?**
> **객체화된 인터페이스마다 각자 다르게 기능하게끔하고 싶다면 어떻게 해야할까?**

1. 과거에는 익명 내부 클래스를 사용하였다.
2. 다음으로는 람다식으로 구현하여 사용한다.

예시를 살펴보자.

## 익명 내부 클래스 📝

```java
// 과거엔 익명 내부 클래스를 사용했었다.
RunSomething runSomething = new RunSomething() {
	@Override
	public void doIt() {
	    System.out.println("hello");
	}
};
```

## 람다식 📝

람다식은 함수형 인터페이스로 정의되어있는 것을 람다 문법으로 사용하는 것을 말하는 것이다. 함수형 인터페이스를 활용하는 것이 람다식인 것이다.

가령 stream()의 map()은 아래와 같이 정의되어있다.

```java
<R> Stream<R> map(Function<? super T, ? extends R> mapper);
```

이제 정의되어있는 함수형 인터페이스를 어떻게 줄이는지를 보자

```java
// 만약 하나의 action이면 다음과 같이 줄일 수 있다.
RunSomething runSomething = () -> System.out.println("hello");

// 만약 여러 action이면 다음과 같이 작성할 수 있다.
RunSomething runSomething = () -> {
    System.out.println("hello");
    System.out.println("Lambda");
    System.out.println("hello");
};


// 만약 외부 변수나 객체 내부 변수와 함수가 coupling,depend 되어있다면 이는 pure function이 아니다.
// (O)
RunSomething runSomething = (number) -> {
    return number + 10;
};
// (X) :: 클래스 외부 변수와 coupling
int baseNum = 10;
RunSomething runSomething = new RunSomething() {
    @Override
    public int doIt(int number) {
        return number + baseNum;
    }
}
// (X) :: 메서드 외부 변수와 coupling
RunSomething runSomething = new RunSomething() {
    int baseNum = 10;

    @Override
    public int doIt(int number) {
        return number + baseNum;
    }
}

```

## 어떻게 사용하는가 ☝️

### 1. 행위를 값으로 취급한다

함수형 인터페이스에서는 하나의 로직을 하나의 값으로서 던지거나 받을 수 있게 된다. Tecoble의 간단한 예제를 살펴보자.

```java
@FunctionalInterface
public interface Validator {
    boolean test();
}
```

```java
public class Main {

    public static void main(String[] args) {
        Random random = new Random();
        int user1 = random.nextInt(10);
        int user2 = random.nextInt(10);
        boolean check = is(() -> user1 < user2);
    }

    public static boolean is(Validator validator) {
        return validator.test();
    }
}
```

위에서 Main 클래스의 is메서드는 Validator 를 인자로 받은 뒤 test를 반환하는 메서드이다. main함수 내의 `boolean check = is(() -> user1 < user2);`을 살펴보면 파라미터 `validator`는 로직인 `() -> user1 < user2`를 파라미터로서 받는다. 즉, 어떤 로직 자체를 값으로 던지거나 받고 있다.

그렇다면 이 로직을 재활용할 수 있을까? **함수형 인터페이스에서는 로직을 던질 수 있기 때문에 가능하다!** 아래와 같이 바꿀 수 있다.

- `() → user1 + user2`
- `() → user == user2`
- `() → user + user2 > 10`

Functional interface에서는 파라미터로 전달하여 사용할 수도 있지만, 직접 객체화해서도 사용가능하다.

```java
Validator validator = () -> user1 < user2;
validator.test();
```

[^3]: 람다식이 캡처하는 외부 변수는 사실상 final(effectively final)이어야 한다. 값이 변경된 변수를 람다 내부에서 참조하면 컴파일 에러가 발생한다. 이는 람다가 멀티스레드 환경에서도 안전하게 동작하도록 하기 위한 설계다.
