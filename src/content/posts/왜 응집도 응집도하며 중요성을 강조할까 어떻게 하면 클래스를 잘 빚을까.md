# Why?

"오브젝트: 코드로 이해하는 객체지향 설계" 4장,5장을 요약함으로서 아래 의문점을 해결한다.

- 낮은 응집도란 무엇이고,
- 낮은 응집도를 가지는 구조의 문제는 무엇이고,
- 어떻게 하면 변경에 유연한 설계를 할 수 있는지,

# What?

## 1.

왜 응집도 응집도하며 중요성을 강조할까?

높은 응집도, 낮은 결합도를 왜 자꾸 강조할까?

사실 응집도나 결합도나 반대말일뿐 같은 개념을 지향하는 단어이다.

이 단어들은 그저 “얼마나 변경이 전염되는가?” 의 기준치를 측정하는 정도값이다. 요컨대 “모듈 A 변경에 따라 모듈 B 가 얼마나 변경되어야하는가?” 이다.

이 정도, 기준치가 중요한 이유는 그저 단 한 가지이다. 이러한 설계가 “변경하기 쉬워서”

이러한 설계를 지향하려면 “각 모듈의 책임” 을 기준으로 설계를 해야한다는 것이다.

그렇다면 책임 기준으로 어떤 과정을 통해 설계해나가야할까?

## 1.

결합도와 응집도 유지방법

### 상태 == 구현

상태를 기준으로 도메인을 설계하게 되는 경우, 상태에 따른 행동이 도출되기에 불안정하다.

객체의 상태는 구현에 속하게 되고, 이 세부사항이 인터페이스에 녹아들면, 캡슐화가 깨지게된다.

이에 따라 변경에 취약할 수밖에 없다.

**내부 상태값 → 세부 구현 → 각각의 여러 인터페이스**

### 책임 == 인터페이스

객체의 책임을 드러내는 인터페이스를 정의함으로써, 세부사항을 숨길 수 있다.

즉, 아래 프로세스를 통해 캡슐화를 지킬 수 있다.

**책임 인터페이스 선언 → 책임에 맞는 세부 구현 → 내부 상태값**

## 3.

낮은 응집도 판별에 따른 클래스 분리법

설계를 하거나 구현을 하다보면 과하게 응집되거나, 너무 분산되어있거나 할 때가 있다. 이러한 판단을 하는 기준치가 있을까?

어떻게 응집도의 낮고 높음을 판단하는가??

### 낮은 응집도 판단 기준

참고자료인 "오브젝트: 코드로 이해하는 객체지향 설계" 에서는 아래와 같이 세 가지 특징을 저술한다.

**하지만 결론부터 이야기하자면, 결국 세 가지 특징 모두 하나로 귀결된다.**

<span style="color:yellowgreen">**바로 "하나의 클래스 당 하나의 책임" 이다. **</span>

(이는 곧 아래 특징1,2,3 중 특징1에 해당한다.)

### 정보전문가

## 5.

변경에 취약한 부분은 합성을 사용

설계를 주도하는 것은 변경이다. 변경에 대해 유연하게 대비하는 방법은 1) 최대한 단순하게 설계 2) 코드 수정없이 유연하게 처리하도록 하는 것이다.

이러한 방법 중에 이펙티브 자바에서도 강조한 개념인 인터페이스화 & 상속보다 합성을 지향 을 사용할 수 있다.

왜 상속은 변경에 취약할까?

아래와 같이 영화 부모클래스와 여러 할인정책에 따른 영화자식이 있다고 하자.

![](/images/velog/7a753f1496cfa6b9.png)

상속을 기반한 위 설계에 따르면 할인정책 추가 및 변경은 다음과 같은 과정으로 처리된다.

1.

영화 인스턴스 생성 2. (변경인 경우) 영화 인스턴스 상태 복사 3.

상태값 및 식별자에 대한 처리 메서드

어떻게 하면 이러한 프로세스를 하나로 묶을 수 있을까?

1.

변경되는 요소를 따로 분리한 뒤, 이를 인터페이스화 2.

### 그렇다고 도메인 서비스를 애그리거트에 주입하지는 말자.

DI 에 빠져있으면 서비스 레이어를 의존하는 것을 보자마자 DI 를 통해 주입하려 할 것이다. 도메인 애그리거트에서도 이러한 강박관념이 투영될 수 있다.

가령 아래와 같이 말이다.

```java
public class Order {
		@Autowired
		private DiscountCalculationService discountCalculationService;
		,,,
}
```

하지만 저자 최범균님은 “도메인 주도 개발 시작하기 : DDD 핵심 ,,,” 책에서 이러한 설계에 대해서 부정적인 견해를 내놓으신다.

> 도메인 객체는 필드(프로퍼티)로 구성된 데이터와 메서드를 이용해서 개념적으로 하나인 모델을 표현한다.

모델의 데 이터를 담는 필드는 모델에서 중요한 구성요소다.

> 그런데 discountCalculationService 필드는 데이터 자체와는 관련이 없다.

Order 객체를 DB에 보관할 때 다른 필드와는 달리 저장 대상도 아니다.

> 또 Order가 제공하는 모든 기능에서 discountCalculationService를 필요로 하는 것도 아니다.
>
> 일부 기능만 필요 로 한다.

일부 기능을 위해 굳이 도메인 서비스 객체를 애그리거트에 의존 주입할 이유는 없다.

> 이는 프레임워크의 기능을 사용하고 싶은 개발자의 욕심을 채우는 것에 불과하다.

나 또한 이러한 의견에 동의하는 바이다. 애그리거트는 엔티티 레이어에 적합하다.

나는 개인적으로 애그리거트는 그저 엔티티와 그에 관련된 로직을 제공하는, “도메인 데이터 파사드 개념의 인스턴스” 역할이라고 생각한다.

도메인 서비스는 이와 달리 응용 레이어와 엔티티 레이어 사이에 존재해야한다고 생각한다. 이러한 역할에 대해서 도메인 서비스가 주입되어야할 이유도 없고, 레이어의 위치도 다르다고 생각한다.

이러한 생각을 기반으로, 위 코드를 수정하자면 이렇게 고칠 것 같다.

    위와 같은 단점들로 인해 개발 과정도 비효율적이고, 다른 사람이 수정하기에도 복잡하고 번거롭게 된다.

어떻게 해결할 수 있을까?

그냥 메서드와 DTO 들을 유스케이스 기준으로 분리하면 된다.

각 메서드와 각 DTO로 분리를 통해 아래와 같이 바꾸면 해결된다.

    ```java
     @CustomExceptionDescription(SwaggerResponseDescription.EXCEL_DOWNLOAD)
     @Operation(summary = "Member 도메인 멤버 리스트 조회 Excel File 다운로드", description = "Member 도메인의 멤버 리스트 Excel File 을 다운로드한다.")
     @GetMapping("/api/member/download-excel/all")
     public ResponseEntity<?> downloadExcel(
         @Parameter(name = "조건파라미터", description = "멤버 리스트 조회 조건파라미터")
         @ModelAttribute MemberAllExcelCondition memberAllExcelCondition
     ) {
         ExcelCreateData excelCreateData = new ExcelCreateData();
         strategy.injectAllMembersTo(excelCreateData, memberAllExcelCondition);
         return buildExcelResponse(excelCreateData, fileService, log);
     }

     @CustomExceptionDescription(SwaggerResponseDescription.EXCEL_DOWNLOAD)
     @Operation(summary = "Member 도메인 상세 정보 Excel File 다운로드", description = "Member 도메인의 상세 정보 Excel File 을 다운로드한다.")
     @GetMapping("/api/member/download-excel/{Id}")
     public ResponseEntity<?> downloadExcel(
         @Parameter(name = "조건파라미터", description = "멤버 상세정보 조건파라미터")
         @ModelAttribute MemberDetailExcelCondition memberDetailExcelCondition
     ) {
         ExcelCreateData excelCreateData = new ExcelCreateData();
         strategy.injectDetailMembersTo(excelCreateData, memberDetailExcelCondition);
         return buildExcelResponse(excelCreateData, fileService, log);
     }
    ```
