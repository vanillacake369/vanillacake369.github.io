# 오잉 그럼 참조변수는?

---

아래와 같은 코드가 있다 하자. 호출순서와 힙과 스택이 어떻게 돌아가는지 그려지는가?

```java
public void m2(int b){
		boolean c;
		m3();
}
public void m3(){
		Account ref = new Account();
		// more code
}
Account {
     Int p;
     Int q;
}
```


Heap에는 아래와 같이 p,q를 가진 Account객체가 생성될 것이다.

![](/images/notion/3e32e1fd2b8b30af.webp)

Stack에는 이 Account객체를 refer하는 reference variable이 생성된다.

![](/images/notion/bff71637f028d53a.webp)

이제 Account를 참조하고 있던 m3가 완료되면, Stack영역에서 m3가 flush-out된다.
이 때, Account를 참조하고 있던 참조변수도 삭제된다.
Account를 참조하고 있는 참조값이 사라졌으므로, Heap에 있는 Account는 GC의 대상이 된다.

# Object LifeCycle

---

## Birth :: Saying `new` :: Constructor Chaining

---

모든 객체는 Object로부터 상속(확장)된다.
superclasses로부터 상속된 subclasses들은 superclasses의 모든 것을 가지고 있다.
따라서 객체가 new에 의해 생성될 때에는, 본인의 instance 뿐 아니라 inheritance tree에 있는 superclasses의 instance까지 생성한다.

> The Big Point there was that every object holds not just its own declared instance variables, but also everything from its superclasses (which, at a minimum, means class Object, since every class extends Object).
> So when an object is created (because somebody said new; there is no other way to create an
> object other than someone, somewhere saying new on the class type), the object gets space for all the instance variables, from all the way up the inheritance tree.


따라서 new를 통해 객체가 생성된다면, superclass의 생성자들을 연달아 호출하는 일을 하는 것이다.

> All the constructors in an object’s inheritance tree must run when you make a new object.


Note해야할 것은, abstract class도 contructor를 가지고 있기 때문에, — abstract class 객체를 생성할 수는 없지만 — 그것의 concrete subclass를 생성하게 된다면, 그것의 super class인 abstract class의 constructor가 호출된다.

여기서 또 주의해야할 점은 어떤 순서대로 진행되느냐이다.

subclass가 존재하기 위해서는 superclasses들이 존재해야하기 때문에 
모든 superclass part가 완전히 형성되어야, 즉 superclass들의 생성자 호출이 완전히 종료되어야 subclass가 생성될 수 있다.
즉 아래와 같이 진행될 것이다.


> Saying new is a Big Deal. It starts the whole constructor chain reaction. And yes, even abstract classes have constructors. Although you can never say new on an abstract class,  an abstract class is still a superclass, so its constructor runs when someone makes an instance of a concrete subclass.
> The superclass constructors run to build out the superclass parts of the object.
> Remember, a subclass might inherit methods that depend on superclass state (in other words,
> the value of instance variables in the superclass).
> For an object to be fully formed, all the superclass parts of itself must be fully formed, and that’s why the superclass constructor must run


이에 따라, 항상 생성자 내부에서는 super()가 가장 먼저 호출되어야 한다.

![](/images/notion/bd0f7f403d535b01.png)

만약 this()를 통해 동일한 클래스 내부에 overloading된 생성자를 호출하고 싶다면, super()를 사용할 수 없다. 무조건 overloading된 생성자만을 this()로 호출해야만 한다.

> Use this() to call a constructor from another overloaded constructor in the same class.
> The call to this()  can be used only in a constructor, and must be the first statement in a
> constructor.
> A constructor can have a call to super() OR this(), but never both!



## Death :: When refer dies, it dies

---

**참조변수가 죽어버리면, Heap에 남아있게 되는 객체에 대한 메모리는 GC가 수거해간다.**

> An object’s life depends entirely on the life of references referring to it. 


그럼 언제 참조변수가 죽냐고?? 
이 참조변수가 local variable이냐, instance variable이냐에 따라 다르다.

- local variable
- instance variable




# 객체 죽이기 :: 세 가지 방법

---

Three ways to get rid of an object’s reference:

1. The reference goes out of scope, permanently
2. The reference is assigned another object
3. The reference is explicitly set to null
